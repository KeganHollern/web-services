# Blocking Scripted Remote Execution

Recently, there have been a lot of server file leaks.
The way cheaters obtain server files has always been the same — Remote Execute some code up to the server that retrieves data and returns it back to the cheater's client.
On receipt the cheater writes the data into [profileNamespace] or a log file or wherever else they like.

After enough time on both sides of this, I've come to think the most effective use of an RE isn't pulling files at all — it's installing some sort of unloggable backdoor.
A way to push messages up to the server without being at the mercy of the [Scripts.txt] filter.
That's where _markers_ come in.

Setting the text of a marker broadcasts a string to every connected client without tripping any of the script filtering.
The cheater seeds a long-running thread on the server using whatever loggable RE they can land — `createUnit`, `addMPEventHandler`, and friends — and that thread watches a specific marker for text changes and executes whatever shows up.
The expensive, traceable part happens once; everything after is invisible.

---

Without going further into how the RE itself lands, here's an approach I came up with to break the marker side of it:

```sqf
//--- lystic's re blocker (this blocks the RE that caused the mass server file leak)
{
    if (markerAlpha _x == 0 && markerText _x == "") then {
        _x setMarkerText "RTEXT = true;";
        uiSleep 1;
        if (missionNamespace getVariable ["RTEXT", false]) then {
            deleteMarker _x;
            //--- Prevent marker from reappearing ?
        };
    };
} forEach allMapMarkers;
```

It walks every map marker and looks for ones that are invisible and have empty text — exactly the shape of a marker being used as a covert channel.
Each candidate gets its text overwritten with `RTEXT = true;`.
If the cheater's server-side thread is doing what I think it's doing, it will execute that string and flip `RTEXT` on the mission namespace.
After a one-second pause we check the variable, and if it came back true the marker was actively being used and we delete it.

---

This isn't finished.
It needs to reset `RTEXT` between iterations, and it needs a way to actually log when an RE has been detected so you have something to act on.
But as a first cut against the technique, it's enough to break the loop.

That's it. Stay safe.
