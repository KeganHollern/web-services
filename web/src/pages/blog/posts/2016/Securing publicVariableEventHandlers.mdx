# Securing publicVariableEventHandlers

A big issue with most game modes in Arma 3 is how they handle network messages.
Most servers I've come across do nothing to protect [`publicVariableEventHandler`] and [`remoteExec`] calls.
There are plenty of ways to lock these down — what follows is just the one I landed on.
I wanted something that could handle multiple concurrent calls without variable collisions, and that didn't leave a simple direct path to the underlying event.

The flow boils down to four steps:

1. Client sends a request to the server.
2. The server asks the target of the request whether they really made the call.
3. The client confirms it sent the request and tells the server to go ahead.
4. The server runs the event code.

I'll walk through how it actually works at the end.

---

Server side:

```sqf
"Suicide" addPublicVariableEventHandler {
    _unit = _this select 1;
    _var = "Suicide_" + str(floor(random(10000)));
    _var addPublicVariableEventHandler {

        //--- the actual event handler code here
        (_this select 1) setDamage 1;

    };
    missionNamespace setVariable ["Suicide2", _var];
    (owner _unit) publicVariableClient "Suicide2";
};
```

Client side:

```sqf
"Suicide2" addPublicVariableEventHandler {
    _var = _this select 1;
    if (AmSuiciding) then {
        missionNamespace setVariable [_var, player];
        publicVariableServer _var;
    };
};
```

And here's how the client kicks off a request to kill itself:

```sqf
AmSuiciding = true;
Suicide = player;
publicVariableServer "Suicide";
```

---

The trick is that the server creates a randomized event handler whose name is only known to the targeted client. Step by step:

1. **On the client:** toggle `AmSuiciding` so the callback knows we really did mean to run this event.
2. **On the client:** `publicVariable` `Suicide` to the server, containing our unit.
3. **On the server (inside the `Suicide` event):** generate a random string to act as our randomized event handler.
4. **On the server (inside the `Suicide` event):** create an event handler attached to that randomized string with our actual event code inside it.
5. **On the server (inside the `Suicide` event):** notify the targeted client of the randomized string.
6. **On the client (inside the `Suicide2` event):** if `AmSuiciding` is true, run the code that notifies the callback on the server.
7. **On the client (inside the `Suicide2` event, inside the `if`):** take the random string the server sent us and create a variable containing our unit object.
8. **On the client (inside the `Suicide2` event, inside the `if`):** `publicVariable` that randomized string so the actual event on the server triggers.
9. **On the server (inside `Suicide`, inside the created event handler):** kill the unit that ran `Suicide`.

This stops anyone from writing a "kill target" script that calls the `Suicide` event directly — they don't know the randomized name, and the targeted client only forwards the request when it actually initiated it.
Using an event handler for suicide is admittedly silly, but the same shape works fine for anything more interesting like admin menu items.

Anyway, that's it. Stupid example, useful pattern.
