# How do Anticheats work?

A lot of people don't understand how scripted anticheats like [Infistar AntiHack] actually work, but the idea is simple.
The whole thing boils down to three tricks: randomizing variable names, leaning on local scope, and pushing heartbeat checks to clients in a way that catches anyone who tries to silence them.

---

The first trick is randomization.
Any global variable that another thread can reach is a variable a cheater can clobber, and if they can change your code they can stop your code from running.
So most anticheats name their entry points with a random string generated at startup:

```sqf
_randSTR = {
    params ["_count"];
    private ["_str", "_min", "_max"];
    _str = [];
    _min = (toArray "A") select 0;
    _max = (toArray "Z") select 0;
    for "_i" from 1 to _count do {
        _str pushBack floor (_min + random (_max - _min));
    };
    toString _str
};
```

So how does a random name protect anything?
In Arma 2 there was no way to enumerate every variable in a namespace without a memory cheat — if the scripter didn't know what your variable was called, they couldn't touch it.
Arma 3 broke that assumption with [allVariables], which makes randomization useless on its own.
The fix is [compileFinal]: it stops the compiled value from being overwritten at the script layer.
Together they push the attack back into the memory-cheat realm.

You can take it further by storing variables on a random object via [setVariable] and [netId], so even the namespace they live on is unknown.
I use this function to pick a random building on any map (default or modded) the server is running:

```sqf
_getBuilding = {
    private ["_building", "_loop", "_wuaioghdwaoih", "_location"];
    _building = objNull;
    _loop = 0;
    while {isNull _building && !alive _building} do {
        _location = locationPosition (nearestLocation [[random (worldSize), random (worldSize), 0], "CityCenter"]);
        _wuaioghdwaoih = nearestObjects [_location, ["House", "Building"], 300];
        if (count _wuaioghdwaoih > 0) then {
            _building = _wuaioghdwaoih select floor (random (count _wuaioghdwaoih));
        };
        _loop = _loop + 1;
        if (_loop == 15000) exitWith { diag_log "Loop exiting @ 15k iterations" };
    };
    diag_log str (_building);
    _building allowDamage false;
    _building;
};
```

The building's net id is a globally unique handle, so the server and clients can both reach it without ever naming it in a global.
Here's the ban function I push to clients — it uses a randomized ban variable and the random object's net id to talk back to the server:

```sqf
_ban = {
    _reason = _this;
    (objectFromNetID '" + _netID + "') setVariable ['" + _banvar + "_' + (getplayeruid player) + '_reason', _reason, true];
    (objectFromNetID '" + _netID + "') setVariable ['" + _banvar + "_' + (getplayeruid player), true, true];
};
```

---

The second trick is staying local.
Locals in SQF leak up scopes but not down — anything declared in the outer scope is visible to inner scopes, and nothing the inner scopes do can be touched from the outside.
So I declare the sensitive bits at the lowest scope, once:

```sqf
_netID = netID _aoiwyhoi;
_serverPassword = "iauwyhdiuh";
_timeout = 15;
_banvar = [10, true] call _randSTR;
_mmvar  = [10, true] call _randSTR;
```

Then I read them from inner functions like `_buildPacket`, which is called several scopes up from where the variables live:

```sqf
_buildPacket = {
    params ["_code", "_netID", "_var"];
    private ["_packet"];
    _packet = ("");
    if (typename _code == typename "") then {
        _code = compile _code;
    };
    _packet = ("
        diag_log 'client packet';
        _thread = [] spawn {
            _ban = {
                _reason = _this;
                (objectFromNetID '" + _netID + "') setVariable ['" + _banvar + "_' + (getplayeruid player) + '_reason', _reason, true];
                (objectFromNetID '" + _netID + "') setVariable ['" + _banvar + "_' + (getplayeruid player), true, true];
            };
            while {true} do " + str (_code) + "
        };
        (objectFromNetID '" + _netID + "') setVariable ['" + _var + "_' + (getplayeruid player), _thread];
    ");
    _packet;
};
```

Because `_netID` and `_banvar` were declared in the outermost scope, nothing else on the client can read or replace them.

---

The third trick — and the one that does most of the work — is the heartbeat.
Honest clients have no reason to stop your code from running.
Cheaters almost always do, because they need to silence the loop that's looking for their cheat.
So the heartbeat sends checks to clients every 10 to 25 seconds, then comes back later to see if the thread it spawned is still alive.
If it isn't, the player killed it, and the player gets banned.

```sqf
diag_log "ANTICHEAT: starting heartbeat";
while {true} do {
    waitUntil {count (allplayers) > 0};
    //--- Randomize data order
    _checkData = [_checkData] call _shuffle;
    //--- Send data
    {
        _var  = _x select 0;
        _code = _x select 1;

        _packet = [_code, _netID, _var] call _buildPacket;
        [_packet] call _AntiCheatRE;
        uiSleep 0.1;
    } forEach _checkData;

    uiSleep (10 + random 15); //--- Random heartbeat delay

    {
        _var = _x select 0;
        [_netID, _var] call _cleanUp;
    } forEach _checkData;
};
```

`_buildPacket` from earlier wraps every check in a `spawn` and writes the thread handle to the random object via `setVariable`.
The heartbeat round-trips through that handle: send code, sleep a random amount, check whether the spawned thread is still there.
Cheaters who blanket-kill threads to defeat one check end up tripping every other check at the same time.

---

The whole thing fits in about 400 lines of SQF including the admin menu, with only three randomized variable names, and it holds up against the same memory-cheat threat model as the commercial brands.
None of it is clever — just three small ideas stacked.

Anyway, that's it. Stay paranoid.
