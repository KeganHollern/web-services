# C# Arma Extension command parsing

I spent a good portion of time last year working with server extensions in [Arma 3].
I wanted a simple way to interface the script and the extension, and to make adding functions into the extension as easy as possible.
What I came up with is a tiny string protocol — `Command|Parameter1~Parameter2~Parameter3` — and a pair of classes that hide the parsing entirely.
The whole `RVExtension` entry point ends up at five lines.

This is pulled from a project I'm actively working on, so a few helper functions called below aren't shown.

---

```csharp
[DllExport("_RVExtension@12", CallingConvention = System.Runtime.InteropServices.CallingConvention.Winapi)]
public static void RVExtension(StringBuilder output, int outputSize, [MarshalAs(UnmanagedType.LPStr)] string function)
{
    outputSize--;

    CommandRequest request = new CommandRequest(function);
    Command command = Command.getCommand(request.Command);
    string result = command.execute(request.Parameters);

    output.Append(result);
}
```

`CommandRequest` parses the raw string into a command name and a parameter array.
`Command.getCommand` looks up the right `Command` subclass for that name (or returns a no-op base instance), and `.execute` runs it.
The result is written back into the buffer Arma handed us.

---

`CommandRequest` is essentially a glorified split.

```csharp
class CommandRequest
{
    public string Command { get; }
    public string[] Parameters { get; }
    public CommandRequest(string input)
    {
        string[] parts = input.Split('|');
        Command = parts[0];
        if (parts.Count() == 2)
        {
            List<string> p = parts[1].Split('~').ToList<string>();
            while (p.Remove("")) { }; //--- remove all black entires
            Parameters = p.ToArray();
        }
    }
}
```

The constructor splits on the pipe, keeps the first half as the command name, then splits the second half on `~` and uses `List.Remove("")` in a `while` loop to chew through any blank entries before handing the rest back as an array.

---

The `Command` base class is only a little more interesting.

```csharp
class Command
{
    public static Command getCommand(string input)
    {
        switch(input)
        {
            case "updatePlayers":
            {
                return new UpdatePlayers();
            }
        }
        return new Command();
    }

    public virtual string execute(string[] parts)
    {
        return "";
    }
}
```

`getCommand` is a flat `switch` over the command name that returns an instance of the matching subclass, or a bare `Command` as a no-op fallback.
`execute` is virtual so each subclass can override it with whatever it actually needs to do.

---

Adding a new command is then just a subclass and an override.

```csharp
class UpdatePlayers : Command
{
    public override string execute(string[] parts)
    {
        //--- Send online player list to server
        if(parts.Count() > 0)
        {
            Globals.PController.updatePlayerInfo(parts);
        }

        PlayerInfo[] players = Globals.PController.getOnlinePlayers();

        //--- parse playerinfo into a webrequest and send api request

        return API.request(APIRequestType.updatePlayers, "");
    }
}
```

Now when `"updatePlayers|Player1"` comes in via [callExtension], `UpdatePlayers.execute` runs with `{ "Player1" }` as its parameter.
New commands cost a single subclass plus one extra `case` in `getCommand`.

---

Most of my newer extensions inside Arma run off this pattern. I'm by no means great at C#, but I found this simple and easy to work with.

The base project came from Maca134's [ARMAExtCS] repo, which is what I'd recommend starting from if you're going down this road yourself.
